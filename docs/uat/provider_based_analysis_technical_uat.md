# Provider-Based Analysis Technical UAT

## Purpose

Technical UAT validates the provider-based analysis path before live provider-symbol execution,
product UI, asset search, watchlist behavior, scheduled refresh, and final product UAT.

Technical UAT is not final product UAT. It does not require asset search UI, watchlist,
scheduled refresh, or live provider calls in CI. It focuses on whether already-fetched provider
data can produce an understandable, safe, and technically valid Market xG result.

## Current Technical Path Under UAT

`Already-fetched StooqHistoricalResponse -> Stooq normalization -> OHLCV validation -> indicator
calculation -> Trend / Momentum score -> Volume / Accumulation score -> Market xG aggregation ->
markdown report`

Current behavior:

- The orchestrator aggregates exactly two implemented categories: `trend_momentum` and
  `volume_accumulation`.
- Missing future categories are expected and should be disclosed by the report and aggregation
  layer.
- Provider metadata is preserved in `ProviderAnalysisResult`, not inside normalized OHLCV rows.
- The current path returns a successful `ProviderAnalysisResult` or raises
  `ProviderAnalysisError`.

## UAT Roles

Sponsor / Product Owner:

- validates whether the report and result are understandable
- validates whether the output aligns with the Market xG product vision
- decides whether the product can proceed to provider-symbol live execution

QA:

- validates scenario behavior
- validates failure handling
- records acceptance evidence

Engineering:

- validates implementation boundaries
- confirms deterministic execution
- confirms no live network dependency in CI

## Technical UAT Entry Criteria

- MXG-032 committed and pushed.
- MXG-033 committed and pushed.
- MXG-034 committed and pushed.
- Unit checks for provider analysis are green.
- No live provider calls are required for UAT execution.
- Deterministic sample provider responses can be used.
- No unresolved known blocker exists in the provider-analysis orchestrator.
- Safe report disclaimer wording is already generated by the report layer.

## Technical UAT Exit Criteria

- Happy-path provider response produces a `ProviderAnalysisResult`.
- Result includes `provider`, `provider_symbol`, `asset_name`, and `report_date`.
- Result includes Market xG score between 0 and 100.
- Result includes exactly the implemented category scores:
  - `trend_momentum`
  - `volume_accumulation`
- Result includes markdown report.
- Markdown report includes safe disclaimer wording:
  - Market xG is probabilistic decision support.
  - Market xG is not price prediction.
  - This report does not guarantee future returns or outcomes.
- Report makes missing categories visible.
- Invalid or insufficient data produces clear failure behavior.
- No live network dependency is introduced into CI or deterministic checks.
- Sponsor / PO agrees the result is understandable enough to proceed to provider-symbol live
  execution.

## Go / No-Go Decision Model

### PASS

- All mandatory scenarios pass.
- No critical product trust issue is found.
- Sponsor / PO approves moving to provider-symbol live execution.

### CONDITIONAL PASS

- Core analysis works.
- Minor documentation, wording, or evidence issues remain.
- Issues are recorded and can be addressed in follow-up tickets.
- Provider-symbol live execution may proceed if Sponsor / PO accepts the risk.

### FAIL

- Core analysis does not produce a valid result.
- Score or report is misleading or not understandable.
- Missing categories are hidden or unclear.
- Invalid data does not fail clearly.
- Live network dependency is introduced into deterministic checks.
- Provider metadata is lost or misleading.
- Sponsor / PO does not approve moving forward.

## Technical UAT Input Set

Expected deterministic UAT inputs:

- Bullish deterministic provider response:
  - sufficient rows
  - generally rising close prices
  - valid OHLCV semantics
- Weak or downtrend deterministic provider response:
  - sufficient rows
  - generally falling or weakening close prices
  - valid OHLCV semantics
- Sideways / mixed deterministic provider response:
  - sufficient rows
  - mostly sideways or mixed close prices
  - valid OHLCV semantics
- Insufficient history response:
  - fewer than the required row count
- Invalid OHLCV response:
  - for example high lower than low, or open outside high / low range
- Invalid metadata request:
  - empty or whitespace-only `asset_name`
  - empty or whitespace-only `report_date`

These inputs can be generated programmatically. They do not require real market data files or
live Stooq calls. They are for Technical UAT only and are not backtests.

## UAT Scenarios

### UAT-001 - Bullish Happy Path

Purpose:

Validate that provider-based analysis can produce a valid and understandable Market xG result
from a rising deterministic provider response.

Given:

- a deterministic `StooqHistoricalResponse` with at least 260 valid OHLCV rows
- close prices are generally rising

When:

- provider-based analysis is run

Then:

- `ProviderAnalysisResult` is returned
- Market xG score is between 0 and 100
- `trend_momentum` category is present
- `volume_accumulation` category is present
- markdown report is generated
- report includes mandatory disclaimer wording
- missing future categories are visible

PO review notes:

- Report should be understandable without reading source code.
- Score explanation should be clear enough for Technical UAT.
- Output should not imply guaranteed future returns.

Expected evidence:

- observed score
- observed categories
- report excerpt or note confirming disclaimer
- note confirming missing categories are visible

Mandatory: yes

### UAT-004 - Insufficient History

Purpose:

Validate that provider-based analysis fails safely when there is not enough history.

Given:

- a valid `StooqHistoricalResponse` with fewer than 253 rows

When:

- provider-based analysis is run

Then:

- `ProviderAnalysisError` is raised
- error clearly explains insufficient history
- no partial Market xG report is produced

PO review notes:

- System must not produce a score from insufficient data.
- Error should be clear enough to map to product messaging later.

Expected evidence:

- failure message
- confirmation no Market xG score/report was produced

Mandatory: yes

### UAT-007 - Provider Metadata Preservation

Purpose:

Validate that provider identity is preserved at result level.

Given:

- a valid `StooqHistoricalResponse`
- `provider` is `"stooq"`
- `provider_symbol` is set, for example `"AAPL.US"`

When:

- provider-based analysis is run

Then:

- `ProviderAnalysisResult.provider` is `"stooq"`
- `ProviderAnalysisResult.provider_symbol` preserves the input provider symbol
- `ProviderAnalysisResult.asset_name` preserves the provided `asset_name`
- normalized OHLCV rows do not contain provider metadata

PO review notes:

- Provider/source context must not be lost.
- OHLCV rows should remain clean for validator compatibility.

Expected evidence:

- provider value
- provider_symbol value
- asset_name value
- confirmation provider metadata is not inserted into OHLCV rows

Mandatory: yes

- Missing categories must be transparent.

Expected evidence:

- report note confirming missing categories are visible

Mandatory: yes

- it does not present the result as investment advice

PO review notes:

Expected evidence:

- disclaimer confirmation
- notes on any risky wording

Mandatory: yes

### UAT-010 - No Live Provider Dependency

Purpose:

Validate that Technical UAT remains deterministic.

Given:

- Technical UAT execution

When:

- provider analysis is run

Then:

- no Stooq live network fetch function is called
- no external market data file is required
- no live network dependency is introduced into CI

PO review notes:

- Technical UAT and CI should not become flaky because of network/provider availability.
- Live provider execution remains a later milestone.

Expected evidence:

- confirmation no live fetch function was called
- confirmation no external market data file was required
- confirmation no CI live network dependency

Mandatory: yes

## Critical Fail Conditions

Technical UAT must fail if any of these occur:

- Market xG score is outside 0 to 100.
- Expected implemented categories are missing.
- Report does not include mandatory disclaimer wording.
- Missing future categories are hidden.
- Invalid OHLCV data produces a successful result.
- Insufficient history produces a successful result.
- Provider metadata is not preserved in `ProviderAnalysisResult`.
- Live Stooq calls are required for deterministic UAT or CI.
- Output implies guaranteed price prediction or investment advice.
- Sponsor / PO cannot understand what the result means at a high level.

## Known Limitations

- No live provider-symbol execution yet.
- No user-friendly asset search yet.
- No symbol resolution yet.
- No UI yet.
- No watchlist or scheduled refresh yet.
- No persisted analysis status yet.
- Only two Market xG categories are implemented.
- No backtesting or calibration yet.
- No FX conversion.
- No `adjusted_close` decision implemented.
- Not investment advice.
- Not guaranteed prediction.

## Next Milestone After Technical UAT

After Technical UAT passes, likely next implementation steps are:

- provider-symbol based live execution outside CI
- optional manual live verification
- product-level analysis status model
- later user-friendly asset search and watchlist flow
